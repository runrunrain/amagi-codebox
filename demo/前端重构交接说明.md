## 二、demo 总体定位

- **设计风格**：苹果 HIG，优雅克制。浅色背景（#FBFBFD）、白卡、柔和双层阴影、系统蓝（#007AFF）点缀、大圆角、-apple-system 字体栈。
- **核心设计原则**：信息架构与功能按重要性分配面积 > 视觉装饰。核心动作占主区域，次要信息折叠隐藏，避免卡片堆叠突兀。
- **目标技术栈**：Vue 3.2 + Element Plus 2.13 + xterm.js 6.0（demo 用原生 HTML 模拟，重构时替换为该栈）。
- **交互核心**：本应用本质是"在终端会话里与 AI 交互"，故**会话是主舞台**（类 Codex 桌面版工作台），而非"仪表盘 + 启动表单"。

### 5.3 Provider Center（`view-provider`，对应需求 C）

#### 5.3.1 IA 调整（核心）

**需求 C**：4 个平级 Tab（服务提供商 / Claude Code / OpenCode / Codex）。

**demo 调整为两级结构**（体现"Provider 是底层资源、Preset 是基于 Provider 的上层配置"的层级关系）：
- **一级主分段**（pill）：`服务提供商` | `预设`
- **预设**下**二级引擎下划线 Tab**：Claude Code | Codex | OpenCode
- 用**区域标签**（`.pc-zone-label`）强化层级语义：
  - 服务提供商区：「底层资源 · API 凭证与端点 · N 个提供商」
  - 预设区：「启动配置 · 绑定提供商与模型参数」

**层级视觉区分**：一级用 pill 分段（圆角胶囊、灰底容器、active 白底），二级用下划线 Tab（无底色、active 蓝色下划线）——两种形态截然不同，避免被误认为同级。

#### 5.3.2 服务提供商（对应 C2 区块2、C3）

- **工具栏**：筛选胶囊（全部/Anthropic/OpenAI）+ 操作按钮（导出配置、JSON 导入、添加提供商）
- **Provider 卡片网格**（`.prov-grid`，自适应多列）：每卡含
  - 名称 + **A/O 格式徽章**（`.fmt.A` 紫 / `.fmt.O` 绿，按"格式类型"统一着色，非品牌色）
  - Base URL（等宽）+ 默认模型（无值显示"-"）+ 密钥配置状态（绿点已配置/灰点未配置）
- **点击卡片** → 进入 ProviderDetail 详情模式（`.pc-detail`）：
  - 返回按钮（`closeProvider`）
  - 基本信息卡：默认模型、当前有效 Base URL
  - API 密钥卡：状态 + 密钥掩码（`applyKeyMask`，显示/隐藏切换）+ 编辑/删除（删除=危险红）
  - API 格式卡：按 provider 格式动态显示 Anthropic 块（A）/ OpenAI 块（O），各含启用开关 + Base URL（+ Organization 仅 OpenAI）

**对应需求**：C2 区块2、C3 服务提供商按钮、C6 ProviderDetail 详情模式。

#### 5.3.3 预设（对应 C2 区块3/4、C3）

- **Claude Code / Codex 预设**（**统一范式**，共用 `renderPresets`）：
  - 按 Provider 筛选胶囊（全部 + 各 Provider）
  - 添加预设按钮
  - 预设卡片（`.preset-card`）：名称 + 关联 Provider 标签 + 参数徽章（`.param`，模型参数用 `.param.model` 蓝色高亮）
- **OpenCode 预设**（**特殊性，配置文件管理**）：
  - 搜索框 + 可视化/JSON 双模式 segmented + 添加预设
  - 可视化模式：OpenCode 专属卡片（`.oc-card`）
    - 「本机默认配置」项（虚线边框 + 「默认」标签，区别于用户预设）
    - 用户预设：名称 + 预设 key（等宽）+ 描述 + 绑定工作区数徽章 + Provider 绑定标签
  - JSON 模式：深色代码块（`.oc-json`）展示 `~/.config/opencode/config.json` 结构（语法高亮：键名蓝/字符串绿/注释灰），体现"OpenCode 预设 = 配置文件"

**对应需求**：C2 区块3（Claude/Codex 预设）、区块4（OpenCode 预设）、C3 各按钮、C5 设计注意（OpenCode 可视化/JSON 双模式）。

## 九、重构建议与注意事项

1. **先建设计系统**：将 §3.3 token 建为 Element Plus 主题变量 + CSS 变量，组件按 §3.4 封装。格式徽章按"类型"着色（A=紫、O=绿），不要按品牌着色。
2. **保留两级导航范式**：一级 pill + 二级下划线 Tab 的形态区分是刻意设计，勿统一为同一种分段。
3. **保留设置模式**：设置不是路由页，是"侧边栏整体切换 + 返回"的模式，勿改回普通路由。
4. **Provider 与预设的层级**：服务提供商（底层资源）与预设（上层配置）必须分层，勿平级为 4 Tab。
5. **ClaudeCode 与 Codex 插件统一**：两者共用同一渲染范式（已安装/市场 + 列表-详情 + 资源筛选 + 市场搜索排序），Codex 叠加特有（状态横幅/统计/重复诊断/source/pid/能力）。
6. **会话即终端**：会话列表在侧边栏，点会话进终端；勿恢复"仪表盘 + 内嵌终端页"分离的设计。
7. **接口对齐**：§7 数据结构对应后端 Wails 绑定（见需求文档技术约束），重构时用真实接口替换 demo 常量。后端服务层见项目 `internal/`（config/session/pty/plugin/codexplugin/workspace/envcheck 等）。
8. **平台差异**：启动模式、Shell 选项按 `platformCaps` 动态渲染。
9. **验证要求**：前端功能必须浏览器实际交互验证（非仅 API 测试）；终端、设置模式、列表-详情联动、筛选联动均需 E2E 验证。
